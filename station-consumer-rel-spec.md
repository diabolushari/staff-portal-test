# Spec: Station Consumer Relation (GeneratingStation -> Connection)

## 1. Objective

Introduce a versioned relation that attaches consumer connections to a generating station and supports:

- station-side dispatch priority (`consumer_priority_order`)
- consumer-side station preference (`station_priority_order`)
- prosumer rule: primary consumer exists and uses station priority `1`
- immutable historical priority values (no auto-resequencing)
- station-first Laravel/Inertia management contract on Generating Station details page

Target linkage:

- station: `src/main/java/com/kseb/consumerservice/entities/generatingstation/GeneratingStation.java`
- consumer/station connection: `src/main/java/com/kseb/consumerservice/entities/connections/Connection.java`

## 2. Current State

- `generating_stations` exists (versioned with `version_id`, logical id `station_id`, and `connection_id`).
- `connections` exists (versioned with `version_id`, logical id `connection_id`).
- no dedicated station-to-consumer relation currently exists.

## 3. Scope

In scope:

- new DB relation table, constraints, indexes, and migration plan
- new JPA entity/repository/service behavior
- new gRPC contract for create/update/deactivate/list/get
- complete validation and concurrency behavior
- Laravel gateway contract for frontend payload/error normalization
- station-first Laravel/Inertia UI specification (routes, props, components, flows, validations)

Out of scope:

- bulk historical backfill implementation (only plan is included)
- automatic priority normalization/resequencing
- connection-page editing surface for this relation (may be added in later phase)
- drag-and-drop or automatic priority resequencing UX

## 4. Terms and Invariants

- **station**: logical generating station (`generating_stations.station_id`)
- **station connection**: `generating_stations.connection_id` (logical `connections.connection_id`)
- **consumer connection**: the consumer-side `connections.connection_id` linked to a station
- **relation identity**: `rel_id` (stable across versions)
- **relation version**: `version_id` (changes on each update)
- **active row**: `is_current = true`, `effective_end is null`, and `deleted_ts is null`

Invariants:

- For a station, active `consumer_priority_order` values are unique.
- For a consumer connection, active `station_priority_order` values are unique.
- For a station-consumer pair, only one active relation can exist.
- Priority values are positive integers (`>= 1`).
- Removing or editing one relation must not change other relation priority numbers.

## 5. Database Specification

### 5.1 Table

Table name:

- `station_consumer_rel` (under schema `consumer` through current Liquibase `currentSchema=consumer`)

Canonical SQL model:

```sql
CREATE TABLE consumer.station_consumer_rel
(
    version_id              INTEGER GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    rel_id                  INTEGER,
    station_id              INTEGER   NOT NULL,
    station_connection_id   INTEGER   NOT NULL,
    consumer_connection_id  INTEGER   NOT NULL,
    consumer_type_id        INTEGER   NOT NULL,
    consumer_priority_order INTEGER   NOT NULL,
    station_priority_order  INTEGER   NOT NULL,
    effective_start         TIMESTAMP NOT NULL,
    effective_end           TIMESTAMP,
    is_current              BOOLEAN   NOT NULL DEFAULT TRUE,
    created_ts              TIMESTAMP NOT NULL DEFAULT now(),
    updated_ts              TIMESTAMP,
    created_by              INTEGER,
    updated_by              INTEGER,
    deleted_ts              TIMESTAMP,
    deleted_by              INTEGER
);
```

### 5.2 Foreign keys and logical references

- Add DB FK:
    - `consumer_type_id -> parameter_value.id`
- Do not add FK to `connections.connection_id` or `generating_stations.station_id` because both are logical ids in
  versioned tables and are not unique per table row version.
- Enforce station/connection linkage in service validations.

### 5.3 Check constraints

- `consumer_priority_order >= 1`
- `station_priority_order >= 1`
- `effective_end is null or effective_end > effective_start`

### 5.4 Active uniqueness constraints (PostgreSQL partial unique indexes)

```sql
CREATE UNIQUE INDEX uq_scr_station_pri_active
    ON consumer.station_consumer_rel (station_id, consumer_priority_order) WHERE is_current = true AND deleted_ts IS NULL;

CREATE UNIQUE INDEX uq_scr_consumer_pri_active
    ON consumer.station_consumer_rel (consumer_connection_id, station_priority_order) WHERE is_current = true AND deleted_ts IS NULL;

CREATE UNIQUE INDEX uq_scr_station_consumer_pair_active
    ON consumer.station_consumer_rel (station_id, consumer_connection_id) WHERE is_current = true AND deleted_ts IS NULL;
```

### 5.5 Read performance indexes

```sql
CREATE INDEX idx_scr_station_active
    ON consumer.station_consumer_rel (station_id, is_current, consumer_priority_order);

CREATE INDEX idx_scr_consumer_active
    ON consumer.station_consumer_rel (consumer_connection_id, is_current, station_priority_order);

CREATE INDEX idx_scr_rel_id
    ON consumer.station_consumer_rel (rel_id);
```

## 6. Parameter Dependencies

### 6.1 Consumer type (`consumer_type_id`)

Must represent relation type values:

- `PRIMARY`
- `SECONDARY`

Decision:

- Use `parameter_value` entries with stable `parameter_code` values `PRIMARY` and `SECONDARY`.
- Service resolves ids by code at startup or via cached lookup.

### 6.2 Prosumer detection

Decision:

- Station prosumer status is derived from `GeneratingStation.generation_type_id`.
- A station is prosumer when its generation type maps to configured prosumer codes.

Default configured code set:

- `["PROSUMER"]`

If multiple generation type codes represent prosumer in production, configure list in application properties and resolve
via `parameter_value`.

## 7. Versioning and Lifecycle Rules

### 7.1 Create

1. Validate station exists and is active (`effective_end is null`).
2. Validate `station_connection_id == station.connection_id`.
3. Validate consumer connection exists and is active/current in `connections`.
4. Validate priorities are positive.
5. Validate active uniqueness (station priority, consumer priority, station-consumer pair).
6. Apply prosumer rules (Section 8).
7. Insert row with `effective_start = now`, `is_current = true`.
8. Set `rel_id`:
    - first version: `rel_id = version_id` immediately after insert
    - subsequent versions: reuse existing `rel_id`

### 7.2 Update priority

Update is versioned (append-only behavior):

1. Lock current active relation row (`for update`).
2. Validate new priorities and prosumer rule.
3. Close old row:
    - `effective_end = now`
    - `is_current = false`
    - `updated_ts/updated_by` set
4. Insert new active row with same `rel_id`, updated priorities, new `effective_start`.
5. Do not modify any other relation rows.

### 7.3 Deactivate relation

1. Lock active relation row.
2. Apply prosumer primary safety checks (Section 8.3).
3. Close row with `effective_end = now`, `is_current = false`, set delete audit fields.
4. Do not update/resequence other rows.

## 8. Prosumer Rules

### 8.1 Active primary uniqueness for prosumer stations

For prosumer stations:

- exactly one active primary relation is required
- primary relation must have `consumer_priority_order = 1`

### 8.2 Create/update validations

- If `consumer_type_id = PRIMARY`:
    - `consumer_priority_order` must be `1`
    - no other active primary may exist for the station
- If `consumer_type_id = SECONDARY` on prosumer station:
    - operation is allowed only if an active primary already exists

### 8.3 Deactivation validation

- Deactivating the last active primary for a prosumer station is rejected.
- If primary replacement is needed, perform replacement in one transaction:
    - deactivate old primary
    - create new primary with priority `1`

Implementation note:

- provide dedicated service method `replacePrimaryConsumer(...)` for atomic switch.

## 11. Laravel Gateway Contract (for Frontend)

This contract is normative for Laravel/Inertia clients and must remain stable across backend changes.

### 11.1 Payload guarantees

- Station-side list payload (`ListStationConsumers`) must be returned ordered by `consumer_priority_order ASC`.
- Consumer-side list payload (`ListConsumerStations`) must be returned ordered by `station_priority_order ASC`.
- On update/deactivate operations, unchanged relation rows must retain their persisted priority values.
- `Get*` not-found behavior must map to empty/null data payloads and not a hard page error.

### 11.2 Validation and error mapping

Validation response must expose field-level errors for:

- `consumer_priority_order`
- `station_priority_order`
- `consumer_type_id`
- `consumer_connection_id`
- `station_connection_id`

Recommended normalized shape at Laravel gateway boundary:

- `code`
- `message`
- `errors[]` with `field`, `message`, `rejectedValue`

Gateway mapping rules:

- gRPC validation error details map to `errors[field]` for Inertia forms.
- Non-field service errors map to top-level `message` and flash/toast error display.
- Unique-index collisions must be normalized to the same field-level keys as pre-check validations.

## 14. Validation Matrix

- Station not active/not found -> validation error (`station_id`)
- `station_connection_id` mismatch with station -> validation error (`station_connection_id`)
- Consumer connection not active/not found -> validation error (`consumer_connection_id`)
- Priority less than 1 -> validation error on corresponding field
- Duplicate station priority -> validation error (`consumer_priority_order`)
- Duplicate consumer priority -> validation error (`station_priority_order`)
- Duplicate active pair -> validation error (`consumer_connection_id`)
- Prosumer primary not priority 1 -> validation error (`consumer_priority_order`)
- Prosumer secondary without existing primary -> validation error (`consumer_type_id`)
- Deactivate last primary on prosumer station -> validation error (`consumer_type_id`)

## 15. Concurrency Rules

- Use `SELECT ... FOR UPDATE` when mutating an active relation row.
- For create/update conflicts, rely on both:
    - pre-checks in service
    - DB unique partial indexes as final guard
- Convert unique-index violations to structured validation errors.

## 17. Acceptance Criteria

- A station cannot have multiple active consumers with same `consumer_priority_order`.
- A consumer connection cannot have multiple active stations with same `station_priority_order`.
- Removing or editing one relation never changes other rows' priority numbers.
- Prosumer station has one active primary consumer at station priority `1`.
- All mutating operations are versioned and audit fields are set.
- Laravel/Inertia consumers receive stable, field-level validation errors and ordered list payloads.
- Generating Station show page provides station-first relation create/update/deactivate flows.
- Frontend never resequences unchanged relation priorities automatically.
